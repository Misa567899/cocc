# CACOYANNIS LIMITED premium studio website

A greenfield Next.js 15 marketing site for a web design studio, built with App Router, GSAP/ScrollTrigger animations, and Tailwind CSS v4. The site delivers a dark-hero home page with an animated code editor, 3D-tilt service cards, a pinned horizontal-scroll process section, six dynamic service pages with static generation, a contact page, and a consultation modal accessible from multiple entry points. The architecture is clean: server components for page shells, client components for interactive sections, a shared context for the consultation modal, and a centralised animation library.

**Watch for:** The custom cursor attaches listeners to a stale snapshot of interactive elements and never updates (confirmed). The word-reveal animation pattern destructively empties `textContent` before scroll-triggered playback, leaving headings blank for users with JS disabled or slow connections (confirmed). The Process section's horizontal scroll has no reduced-motion fallback â€” content is clipped with no way to reach it (confirmed). `dangerouslySetInnerHTML` in `DesignShowcase` accepts compile-time static strings so the XSS surface is zero, but the pattern sets a precedent that could widen later (possible).

## High-level view

The layout hierarchy is well-structured: a root layout provides font loading, metadata, skip-to-content, the consultation context, custom cursor, navigation, and smooth scroll wrapper. Pages are server components that compose client-side section components. Static params generation for services means all six pages are pre-rendered at build time.

Animation is the dominant concern. Every section registers its own GSAP context and ScrollTrigger instances, cleans up via `ctx.revert()`, and several (Hero, Process) check `prefers-reduced-motion` early. The pattern of splitting text nodes into spans for word/character reveals is repeated verbatim in at least seven components rather than being abstracted through the existing `animateWordReveal` utility in `lib/animations.ts`.

The consultation flow uses React Context at the root, a modal with GSAP open/close animations, and a form that ultimately produces a `mailto:` link. There is no server-side form handling; the "submit" action opens the user's mail client with pre-filled content, showing a success state regardless of whether the mail client opened.

The cursor implementation has a lifecycle gap: it queries interactive elements once on mount and never re-queries after route changes or dynamic DOM updates.

## Custom cursor lifecycle gap

`CustomCursor.tsx` runs a single `useEffect` on mount that calls `document.querySelectorAll("a, button, [data-cursor-hover]")` and binds `mouseenter`/`mouseleave` handlers to the resulting NodeList. In a Next.js App Router application with client-side navigation, the DOM mutates after route transitions, the mega menu inserts links dynamically, and the consultation modal mounts/unmounts form elements.

None of these post-mount elements receive the cursor hover effect. Elements added after mount are permanently orphaned from the cursor system. The cleanup function only removes listeners from the initial set, so if elements in that set are destroyed by React (mega menu links re-rendering), their listeners are cleaned up â€” but replacements get nothing.

The fix is event delegation on `document` (check `e.target.closest("a, button, [data-cursor-hover]")` in the mouseover handler) or re-running the query after route changes via `usePathname()`.

## Word-reveal text destruction and reduced-motion inconsistency

The word-reveal pattern used across Hero, DesignShowcase, Services, Process, Footer, ServiceHero, ServiceCTA, and ContactCTA works by reading `element.textContent`, setting it to `""`, creating `<span>` wrappers programmatically, then animating them. The text is destroyed during the first render effect â€” if JS errors or hydration is slow, headings stay empty.

The `prefers-reduced-motion` guard is inconsistently applied:
- **DesignShowcase, Process, ServiceContent**: return early before text manipulation (correct)
- **Hero**: sets opacity to 1 but still destroys and rebuilds text
- **Services, Footer, ServiceCTA, ContactCTA, ContactHero**: no guard at all

A resilient approach: render text in JSX so it's visible in server-rendered HTML, apply a CSS class (`clip-path: inset(0 0 100% 0)` or `opacity: 0`) as the initial hidden state, and have GSAP animate that off. The text is always present in the DOM; animation is purely visual enhancement.

## Process horizontal scroll reduced-motion fallback

With reduced motion enabled, the GSAP context in `Process.tsx` returns early at line 78. The ScrollTrigger pin never initializes, so the section renders with its static layout: `h-screen flex-col justify-center overflow-hidden`. The track inside has `width: fit-content` and contains 8 cards at 300-380px each (~2700-3040px total). Without the horizontal scroll animation, users see only the first ~3 cards with no way to access the rest â€” the container clips them and there's no `overflow-x: auto` fallback.

```
Container: h-screen, overflow-hidden
  Track: flex, width: fit-content (~3000px)
    Card 01 | Card 02 | Card 03 | [clipped] Card 04...08
```

Add a CSS baseline: `@media (prefers-reduced-motion: reduce) { .process-track { overflow-x: auto; } }` or restructure to a vertical stack when animations are disabled.

## Modal and mobile menu focus management

`ConsultationModal` and `MobileMenu` handle Escape key dismissal and backdrop click, but neither implements focus trapping. When the modal is open, pressing Tab moves focus to elements behind the backdrop. The `ConsultationModal` doesn't move focus into itself on open or return focus to the trigger on close â€” both required by the WAI-ARIA dialog pattern.

The body scroll lock (`document.body.style.overflow = "hidden"`) is applied in both components. `MobileMenu`'s cleanup in the effect return also resets overflow, but if the component unmounts while `isOpen` is still true (e.g., during navigation), the cleanup fires correctly. However, `ConsultationModal` sets overflow in the conditional branch â€” if `isOpen` changes rapidly (double-click edge case), the GSAP `onComplete` callback that resets overflow could race with the next open.

## Consultation form mailto limitations

The form validates client-side, constructs a `mailto:` URL with subject and body, assigns it to `window.location.href`, then immediately calls `setSubmitted(true)`. The success state renders regardless of outcome. On devices without a configured mail handler, the navigation either fails silently or shows a browser error page, but the React component has already unmounted the form and shows "Thank You."

The success message acknowledges this ("Please complete the email submission to reach our team"), but the UX fails completely when no mail client exists. A detection approach: use `window.open` with the mailto URL and check whether it was blocked, or provide a copy-to-clipboard fallback for the composed message.

## Missing static assets

Referenced assets that don't exist in `/public`:
- `/videos/hero-reel.mp4` â€” used in `<video>` with `preload="none"`, so no request until the browser decides to buffer
- `/images/hero-poster.jpg` â€” used as `poster` attribute, requested immediately on mobile
- `/images/services/*.jpg` (6 files) â€” referenced in `lib/constants.ts` `previewImage` field, but currently not rendered in any `<img>` tag (MegaMenu shows a text placeholder instead)

The poster image 404 will be visible on mobile where the video may not autoplay. The service images are inert today but represent broken data contracts.
