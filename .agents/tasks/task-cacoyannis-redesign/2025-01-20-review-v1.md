# Editorial redesign of CACOYANNIS home page sections

A visual redesign pass that replaces the dark video-hero and code-editor aesthetic with an editorial, white-background, typography-driven experience. The Hero switches to a split-layout with clamp-sized heading and mouse-following decorative shapes. DesignShowcase becomes a scroll-driven browser mockup assembling piece-by-piece, plus a floating UI component grid. Services gains 3D mouse-tracking tilt, per-card gradient overlays, and index numbers. Process gets an accent-line topped card style, dashed step connectors, and a gradient progress bar. Supporting changes span the cursor (touch detection, heading state), MagneticButton (extended area, scale), navigation (scroll shadow), footer (separator, link animations), globals (typography scale clamp refinements, scrollbar, keyframes), and removal of broken `previewImage` paths from constants.

**Watch for:** The Hero's rAF-driven mouse-follow loop runs continuously regardless of visibility or reduced-motion preference — it fires `gsap.to` three times per frame even when the hero is scrolled off-screen (confirmed). The word-reveal pattern destructively empties `textContent` before animations trigger, leaving headings blank for users with JS disabled or connection delays (confirmed, affects Hero/DesignShowcase/Services/Process/Footer). The `MagneticButton` extended-area check always passes when the mouse is already inside the element, making the "isNear" guard a dead branch for `onMouseMove` React events that only fire while inside the element (confirmed). The Process progress bar is fixed-positioned at viewport top and always visible — its `scaleX` is only driven by ScrollTrigger when the Process section is pinned, meaning it appears as a stale bar on initial load and after the pin releases (likely).

## High-level view

The Hero section replaces a broken-video dark overlay with a two-column editorial layout. Three decorative shapes respond to mouse position via a persistent rAF loop that doesn't gate on section visibility or reduced-motion preference after initial load.

DesignShowcase is now a scroll-scrubbed browser-assembly animation followed by a floating card grid. The floating cards run indefinite `yoyo` tweens and scroll-driven parallax targeting the same `y` property — the parallax overwrites the bob on every scrub update, making the vertical float invisible once scrolling begins.

Services introduces per-card mouse-tracking state with GSAP-powered 3D rotation. The pattern keeps GSAP transforms on `Link` elements which also have CSS transitions — two animation systems targeting the same element can conflict if timing overlaps.

The Process section pins itself and scrolls cards horizontally with a fixed-position gradient progress bar at the top of the viewport. The bar's fixed positioning means it's unconditionally visible regardless of which section the user is in.

MagneticButton's 20px extended detection area is bound via `onMouseMove` (only fires inside the element), so the proximity check is dead code.

<details>
<summary>Issues (5)</summary>

1. **Hero rAF loop ignores visibility and reduced-motion** — The `requestAnimationFrame` loop in Hero runs continuously from mount to unmount, calling `gsap.to` three times per frame even when the section is scrolled off-screen or the user prefers reduced motion. Gate on an `IntersectionObserver` or at minimum check `prefersReducedMotion` before entering the loop.
2. **Word-reveal leaves headings blank without JS** — Five components destructively clear `textContent` to build span wrappers. If JS execution is delayed or disabled, headings render empty. Consider using CSS `clip-path` / `overflow:hidden` on server-rendered markup and animating children without removing the original text node, or rendering the words as React elements instead of DOM manipulation.
3. **MagneticButton extended-area check is unreachable** — `onMouseMove` is a React synthetic event that only fires while the pointer is inside the element's bounds. The 20px padding check will always pass. To respond outside the element, the listener needs to be on `window` or a parent, or use `onPointerMove` with pointer capture.
4. **Process progress bar always visible** — The `<div ref={progressRef}>` is `position: fixed; top: 0` with `z-30`. It renders a 4px accent-gradient bar at the top of the viewport permanently. Outside the pinned scroll range its `scaleX` stays at whatever value ScrollTrigger last set (0 on initial load, 1 after full scroll). Add conditional visibility tied to the pin state.
5. **Competing animation systems on ServiceCard** — GSAP drives `rotateX`, `rotateY`, `scale` via `gsap.to` while Tailwind's `transition-shadow duration-300` also runs on the same element. If the user moves the mouse in and out rapidly, GSAP's ongoing tween and CSS transition target overlapping properties (`transform`, `box-shadow`), potentially causing visual glitches. Pick one system per property.

## Hero: rAF loop lifecycle

The mouse-follow effect in `Hero.tsx` starts a `requestAnimationFrame` loop on mount:

```typescript
const animateShapes = () => {
  const { x, y } = mousePos.current;
  // ...
  gsap.to(shape1, { x: deltaX * 15, y: deltaY * 10, ... });
  gsap.to(shape2, { x: deltaX * -20, y: deltaY * -12, ... });
  gsap.to(shape3, { x: deltaX * 10, y: deltaY * 8, ... });
  rafId.current = requestAnimationFrame(animateShapes);
};
rafId.current = requestAnimationFrame(animateShapes);
```

Three issues compound. The `prefersReducedMotion` check causes an early return that skips the GSAP context block, but the rAF loop is started *outside* that guard — so users with reduced-motion enabled still get the loop running, animating shapes into positions nobody asked for.

There's no visibility gating. Once the user scrolls past the hero, the loop continues calling `gsap.to` on off-screen elements. An `IntersectionObserver` or `ScrollTrigger` toggle would let the loop pause.

The loop calls `gsap.to` unconditionally every frame. GSAP's overwrite logic mitigates redundant tweens, but three tween allocations per frame fire even when mouse position hasn't changed.

## Word-reveal and progressive enhancement

The word-reveal pattern appears in Hero, DesignShowcase, Services, Process, and Footer:

```typescript
const headingText = heading.textContent || "";
heading.textContent = "";  // <-- heading is now empty in the DOM
// ... build spans, animate them in
```

For the normal animation path, the heading is blank in the DOM between the `textContent = ""` call and the GSAP timeline completing. If GSAP's JS fails to load or execute (CDN issue, script error), the heading stays blank permanently. A React-side approach (rendering `<span>` elements from JSX) would be more resilient and easier to type-check.

## DesignShowcase: compound transforms on floating cards

The floating grid items receive two separate GSAP animations:

```typescript
// Parallax (scroll-driven, targeting y)
gsap.to(item, { y: -30 - i * 10, scrollTrigger: { ... scrub: 1.5 } });

// Bob animation (infinite yoyo, also targeting y)
gsap.to(item, { y: "+=8", rotateX: 2, ... repeat: -1, yoyo: true });
```

Both target `y` on the same element. The relative `+=8` in the bob tween is evaluated once at creation time, not relative to the scroll-driven `y`. The scroll parallax tween overwrites `y` to an absolute value on every scrub update. The net effect is that the bob's `y` contribution is swallowed — only `rotateX`/`rotateY` from the bob survives. The vertical float is likely invisible once scrolling begins.

## MagneticButton extended-area logic

```typescript
const handleMouseMove = useCallback((e: React.MouseEvent) => {
  const padding = 20;
  const isNear =
    e.clientX >= rect.left - padding &&
    e.clientX <= rect.right + padding &&
    e.clientY >= rect.top - padding &&
    e.clientY <= rect.bottom + padding;
  if (!isNear) return;
  // ...
}, [strength]);
```

This handler is bound via `onMouseMove` — a React synthetic event that only fires while the pointer is within the element's DOM boundary. `isNear` will always be true. To achieve the "pull from a distance" effect, the listener needs to be attached to a larger transparent overlay or to the window, filtering events by proximity.

## Process progress bar scope

```tsx
<div className="fixed top-0 left-0 right-0 z-30 pointer-events-none">
  <div ref={progressRef} className="h-1 origin-left"
    style={{ transform: "scaleX(0)", background: "linear-gradient(to right, #050FA3, #1a2fd4)" }} />
</div>
```

ScrollTrigger only drives `scaleX` while the Process section is in its pinned scroll range. After the pin releases, `scaleX` stays at 1 and the bar remains visible as a permanent accent line at the top of the page for the rest of the scroll journey. Toggle `opacity` in an `onEnter`/`onLeave` callback, or scope the fixed positioning to only apply during the pin.
